# 骨牌平铺问题

## 简介

一个交互式网页工具：在左边画出一块区域，在中间选好若干「碎块」形状，由程序判断这些碎块能不能**只平移、不旋转**，无重叠地**正好铺满**那块区域。

## 快速开始

### 环境要求

- Node.js 18+
- pnpm
- 任意现代浏览器（Chrome、Firefox、Safari、Edge）

### 运行

在项目根目录执行：

```bash
pnpm install
pnpm dev
```

浏览器访问终端里提示的本地地址（一般为 `http://localhost:5173`）。

生产构建：

```bash
pnpm build
pnpm preview
```

## 功能说明

页面为**固定三栏**：左棋盘、中碎块、右结果，流程不跳转页面。

### 左栏：题目区域

- 棋盘为 **8×8**
- 在棋盘上点出红色区域：**单击**一格为切换（空则画上、有则擦掉），**按住左键拖动**可连续铺色，快速划过时会自动补全线段经过的格子
- 「清空」清除题目

### 中栏：碎块（两步循环）

- 中栏底部 **「清除」** 会清空已添加的碎块并重置格子数选择，不影响左侧题目
- 顶部居中显示 **碎块总格 / 题目格**（如 `9/9`、`2/9`），底色随是否对齐变化
1. **选格子数**：点「1 格」～「5 格」之一；再点一次可收起形状列表
2. **选形状**：下方列出该格子数下所有**固定**连通形状（仅平移视为同一形状；旋转/翻面算不同形状，与「只能平移」规则一致）
3. **点中一个缩略图**，该形状即加入下方「已添加的碎块」列表
4. 需要下一块时：再选格子数（可与上次相同），再点形状，重复即可
5. 列表中可「移除」某块；在列表里点某块可在左侧棋盘高亮对应碎块形状（相对原点）

### 右栏：判断

- 点「开始判断」后，程序会尝试：能不能像拼图一样，把中间栏的碎块**只平移、不旋转**，一块块摆进左侧红区，**正好铺满、互不重叠**。
- 若左侧红格总数和碎块格子总数对不上，会先提示「格子数不一致」，不必等搜索结束。

## 背后的算法思路（尽量说人话）

### 问题在算什么

把红区看成一堆要盖住的格子，把碎块看成几块**形状固定、只能平移**的硬纸片。程序要回答：**是否存在一种摆法**，让每块纸片占住的格子都在红区内、任意两块不叠在同格上，且所有纸片合起来**恰好**盖住红区的每一格（多一格、少一格都不行）。

这其实是一个**组合搜索**问题：合法摆法可能有很多种，也可能一种都没有，计算机只能系统地「试」。

### 第一步：必要条件（能立刻否定的）

若「红区总格数 ≠ 所有碎块格子数之和」，那无论怎么摆都不可能刚好铺满，程序会直接报错，**不用进入搜索**。这是最简单的剪枝。

### 第二步：按什么顺序试碎块

搜索不是随机乱试。实现里会把碎块按**格子数从多到少**排好，**先安排大块、再安排小块**。直觉上：大块能放的位置更少，更容易很快暴露出「根本摆不下」，比先塞小块再发现大块无处放要省事。这不是数学上的唯一法则，但在实践里很常见、也好写。

### 第三步：对「下一块」试哪些摆放

对某一块碎块，「平移」就是整体左右上下挪动。程序需要枚举：**这一块摆在红区里的哪些位置是值得一试的**。实现上的做法是：用形状的一个固定参考角（代码里取该碎块所占格子的左下角一侧）去对齐红区里的**每一个格子**，每一种对齐方式对应一种平移；再检查这一摆法是否整块都落在红区内、是否和已经摆好的块打架。过不了检查的摆法直接扔掉。

你可以把它理解成：先在桌上固定好前几块，然后拿起下一块，在红区里**挨个候选位置**试放，能放稳再接着摆后面的；放不稳就换下一个候选。

### 第四步：回溯（走不通就回头）

程序按碎块顺序深一步浅一步地试：某一步若当前碎块**所有合理位置都试完仍接不上**，就**撤销**上一块的摆放，换上一块的下一种摆法，再继续——这就是**回溯**：像走迷宫，走到死胡同就退回岔路口换一条路。

搜索进行时，中间栏会显示回溯相关信息；其中的回溯次数，就是在统计这种「走进死胡同再退回」发生了多少次。次数多不一定代表算得慢，但说明这道题在搜索过程里比较「绕」。

### 适用范围与代码位置

这种穷举 + 回溯适合本 demo 的**小棋盘、碎块不多**的场景；若将来红区极大、碎块极多，组合会爆炸，就要考虑更专门的算法或约束求解器，那是另一个量级的问题。

具体实现见 **`src/solveTiling.ts`**（判断能否铺满）；界面与形状枚举在 `src/components/`、`src/Utils/` 等处。
